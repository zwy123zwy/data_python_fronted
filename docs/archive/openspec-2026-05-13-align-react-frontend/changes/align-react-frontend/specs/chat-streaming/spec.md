## ADDED Requirements

### Requirement: Chat interface with SSE streaming

The system SHALL provide a chat interface that sends user queries via SSE and renders streaming responses in real-time. Each node output from the backend MUST be displayed as it arrives, grouped by node name (SqlGenerateNode, PythonGenerateNode, ReportGeneratorNode, etc.).

#### Scenario: Send query and receive streaming response
- **WHEN** user types a query and presses Enter
- **THEN** system creates a new session (if none active), saves the user message, opens an EventSource to GET /api/stream/search, and renders each SSE message as it arrives

#### Scenario: Display node output blocks
- **WHEN** SSE data arrives with nodeName and textType
- **THEN** system renders the output in the appropriate format (SQL in code block, JSON formatted, text as-is, result set as table)

#### Scenario: Stream completion
- **WHEN** SSE event "complete" is received
- **THEN** system marks streaming as finished, saves the complete message history, and enables the input field

#### Scenario: Stop streaming mid-response
- **WHEN** user clicks "Stop" button while streaming
- **THEN** system closes the EventSource connection, saves partial messages, and makes the input field editable

### Requirement: Multi-session management

The system SHALL support multiple chat sessions per agent. Users MUST be able to create, switch, rename, pin/unpin, and delete sessions. The active session SHALL be visually highlighted in the sidebar.

#### Scenario: Create new session
- **WHEN** user clicks "New Chat" in the session sidebar
- **THEN** system calls POST /api/agent/:agentId/sessions, creates a new session with default title, and switches to it

#### Scenario: Switch session
- **WHEN** user clicks on a different session in the sidebar
- **THEN** system loads that session's messages via GET /api/sessions/:sessionId/messages and displays them, preserving the previous session's streaming state

#### Scenario: Rename session
- **WHEN** user double-clicks a session title and enters a new name
- **THEN** system calls PUT /api/sessions/:sessionId/rename and updates the sidebar

#### Scenario: Pin/unpin session
- **WHEN** user clicks the pin icon on a session
- **THEN** system calls PUT /api/sessions/:sessionId/pin and reorders the session list (pinned sessions first)

#### Scenario: Delete session
- **WHEN** user clicks delete on a session and confirms
- **THEN** system calls DELETE /api/sessions/:sessionId and removes it from the sidebar

### Requirement: Session title streaming update

The system SHALL listen to an SSE stream for session title updates. When the backend auto-generates a title for a session, the sidebar MUST update in real-time.

#### Scenario: Receive title update
- **WHEN** a session's title is auto-generated by the backend and broadcast via GET /api/agent/:agentId/sessions/stream as "event: title-updated"
- **THEN** system updates the session title in the sidebar without requiring manual refresh

#### Scenario: SSE reconnect on error
- **WHEN** the session title SSE connection drops
- **THEN** system automatically reconnects after a 3-second delay

### Requirement: Message history

The system SHALL load and display historical messages when a session is selected. Messages MUST be rendered in chronological order with role-based styling (user messages on the right, assistant messages on the left).

#### Scenario: Load message history
- **WHEN** user selects a session from the sidebar
- **THEN** system fetches GET /api/sessions/:sessionId/messages and renders all messages

#### Scenario: Save message after stream
- **WHEN** a streaming response completes (event: complete)
- **THEN** system saves the assistant message with messageType (e.g., "html-report", "markdown-report") and metadata via POST /api/sessions/:sessionId/messages

#### Scenario: User message with titleNeeded flag
- **WHEN** user sends the first message in a session
- **THEN** system sets titleNeeded=true in the message metadata to trigger backend title generation

### Requirement: HTML report download

The system SHALL allow downloading an HTML report for a session.

#### Scenario: Download HTML report
- **WHEN** user clicks "Download Report"
- **THEN** system calls POST /api/sessions/:sessionId/reports/html and triggers a browser file download of the HTML blob

### Requirement: Persistent streaming state across session switches

The system SHALL preserve streaming state (in-progress node blocks, report content) when users switch between sessions, ensuring they can return to an active streaming session and see current progress.

#### Scenario: Switch away and back during streaming
- **WHEN** user switches from session A (currently streaming) to session B and back to session A
- **THEN** session A's streaming progress (node blocks, report content) is fully restored; the EventSource remains active in the background

### Requirement: Preset questions in chat

The system SHALL display agent preset questions as clickable chips above the chat input for quick user input.

#### Scenario: Click preset question
- **WHEN** user clicks a preset question chip
- **THEN** the question text is populated into the chat input and automatically sent

### Requirement: Input modes (nl2sql and human feedback toggle)

The system SHALL allow toggling "NL2SQL Only" mode (skip Python analysis and report) and "Human Feedback" mode (enable plan review before execution).

#### Scenario: Toggle NL2SQL Only
- **WHEN** user enables the NL2SQL Only toggle
- **THEN** subsequent queries are sent with nl2sqlOnly=true parameter

#### Scenario: Toggle Human Feedback
- **WHEN** user enables the Human Feedback toggle
- **THEN** subsequent queries are sent with humanFeedback=true parameter and the SSE flow will pause for user approval
